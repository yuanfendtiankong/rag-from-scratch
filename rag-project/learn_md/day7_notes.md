# Day 7 Notes

## 今天学习的内容

今天我主要围绕“让项目完成度更高”和“继续深入理解检索”这两个目标，对 week1 这个最小 RAG 项目做了一轮整理和升级。相比 Day 6 更偏向接入外部大模型，Day 7 的重点更像是在补齐一个学习型项目的结构、可解释性和后续扩展能力。

今天我完成的工作，主要集中在四个方向：

1. 为项目增加统一入口 `app.py`
2. 让规则式回答带上更明确的引用来源
3. 对当前的简化检索方式做了一轮更深入的优化
4. 将项目真正接到本地真实 embedding / 向量检索上

## 我今天做了什么

1. 将原本空的 `app.py` 写成项目的统一入口
2. 在 `app.py` 中加入模式选择，可以在“规则版 RAG”和“New API 版 RAG”之间切换
3. 保留了已有的回退机制，让 New API 调用失败时可以回退到规则版回答
4. 把原来散落在多个脚本里的终端交互入口，整理成了更像“一个应用”的形式

6. 在总结部分补充“主要参考了哪些 chunks”，增强回答的可解释性
7. 将原来的 `retrieve.py` 入口整理为 `retrieve_demo.py`

10. 先尝试了纯 Jaccard 相似度
11. 在实验后发现纯 Jaccard 对“短 query + 长 chunk”的问答检索场景不够理想
12. 最后把检索改成了“query coverage + Jaccard”的混合相似度
13. 将检索分数统一改为 0 到 1 之间的小数，并同步调整了相关输出格式
14. 调整了 `generator.py` 中的拒答阈值，使其和新的分数体系匹配
15. 在 `config.py`、`embedder.py`、`retriever.py` 中为后续接入 `sentence-transformer` 版 embedding 预留了结构
16. 在本地下载 `paraphrase-multilingual-MiniLM-L12-v2` 模型，并改为从本地目录加载
17. 将默认 embedding 模式切换为 `sentence_transformer`
18. 过滤掉像“。”这样的极短无信息量 chunk，避免它们干扰真实 embedding 排序
19. 优化检索过程，让一次请求里只编码一次 query，而不是对每个 chunk 都重复编码 query
20. 在真实 embedding 检索上增加 lexical baseline，做了一个轻量 hybrid retrieval
21. 调整规则版回答生成逻辑，让主回答优先来自 top1 chunk
22. 增加 `data/processed/chunk_embeddings.json`，将 chunk 向量预计算并缓存到本地
23. 让检索流程优先读取本地 chunk 向量缓存，减少重复编码开销

## 我对今天修改的理解

### 1. 统一入口的意义

之前这个项目虽然已经可以运行，但更像一组彼此独立的小脚本。今天补上 `app.py` 以后，我更清楚地理解了“入口层”和“核心逻辑层”的区别：

- `app.py` 负责和用户交互
- `pipeline.py` 负责组织流程
- `generator.py`、`retriever.py`、`embedder.py` 负责具体功能

这让我意识到，一个项目完成度高不高，不只是看“能不能跑”，还要看结构是不是清楚。

### 2. 引用来源的意义

今天我把规则式回答改成了带来源标注的形式，例如：

- `[chunk 1] ...`
- `[chunk 14] ...`

这个改动虽然不大，但让我更强烈地感受到 RAG 和普通聊天系统的区别。RAG 不只是要给出答案，还应该让人知道这个答案是根据哪些资料整理出来的。这种“可解释性”其实是 RAG 的一个很重要的价值。

### 3. 为什么只用 Jaccard 不够

今天一个特别有收获的点，是我没有停留在“把旧打分改成一个更高级的公式”这个层面，而是真正做了一次小型检索实验。

一开始我把分数从“交集大小”改成了 Jaccard 相似度，也就是：

- 交集大小 / 并集大小

理论上这更公平，因为它做了归一化，不会只因为 chunk 更长就天然拿高分。

但在实际测试里，我发现问答场景下经常是：

- query 很短
- chunk 很长

这时纯 Jaccard 会因为并集太大，把本来相关的长 chunk 分数压低，结果反而不如预期。

这让我意识到：检索方法不是“数学上越正规越好”，而是要结合具体任务来选。

### 4. query coverage 的意义

后来我加入了 `query_coverage()` 这个指标，它表示：

- 用户问题里的字符片段，有多少真的在 chunk 中出现了

这个思路非常适合问答型检索，因为问答检索更关心：

- 这个 chunk 有没有覆盖住我的问题

而不只是关心“query 和 chunk 整体长得像不像”。

所以最终我用了混合分数：

- 0.7 * query coverage
- 0.3 * Jaccard

这个改动让我更理解了一件事：

embedding / retrieval 的优化，很多时候不是简单换个模型，而是先把“表示”和“相似度”的设计想清楚。

### 5. 真实 embedding 最终接入成功

今天 Day 7 的后半段，我最终不只是“预留结构”，而是真正把本地真实 embedding 检索接进了项目主链路：

- 在 `config.py` 中增加了 embedding 模式配置
- 在 `embedder.py` 中预留了 `sentence_transformer` 模式
- 在 `retriever.py` 中预留了余弦相似度函数
- 将 Hugging Face 模型手动下载到本地 `models/` 目录
- 让 `SentenceTransformer` 从本地路径加载，而不是运行时再联网下载

这一步让我更理解“模型依赖完整落地到本地”和“只是 pip 安装成功”并不是一回事。真正要让一个本地 embedding 方案稳定跑起来，还必须把模型文件本身准备好。

### 6. 为什么真实 embedding 接入后还要继续调优

在本地模型真正接起来之后，我很快发现：

- 功能通了，不代表结果自动更好
- 真实 embedding 检索第一次跑出来时，`RAG是什么？` 虽然能检索，但排序和最终规则版回答都不够理想

后来我逐步做了几件更贴近真实检索系统的优化：

1. 过滤无信息量短 chunk
2. 一次检索只编码一次 query
3. 用 `dense + lexical` 做 hybrid retrieval
4. 让规则版主回答优先来自 top1 chunk
5. 预计算并缓存所有 chunk 的向量

这些步骤让我很明确地感受到：

- 真实 embedding 只是开始
- 检索质量优化本身就是一个独立问题
 - 检索一旦可用，性能问题很快也会变成必须处理的工程问题
相比只停留在“能跑”，这一轮修改让我更接近真正理解“检索系统为什么需要多层调优”。

## 我的实验观察

### 问题 1：RAG 是什么？

在今天的检索实验中，我用 `RAG是什么？` 做了对比。

一开始纯 Jaccard 版本的结果并不理想，定义所在的 chunk 没有稳定排在第一位。后来改成“文本规范化 + query coverage + Jaccard”之后，`chunk 1` 重新回到了第一名，结果更符合直觉。

这说明：

- 检索效果不是只靠一个 fancy 的公式就能自动变好
- 具体任务场景会影响相似度设计

### 问题 2：回答的可解释性

在今天之前，系统虽然也会打印参考 chunks，但回答正文和参考 chunks 之间的关系不够紧。改完以后，我能在回答正文里直接看到：

- 这句话来自哪个 chunk
- 总结主要参考了哪些 chunk

这让我感觉现在的输出更像一个真正的 RAG 小系统，而不是简单拼接文本。

### 问题 3：真实 embedding 初版为什么还不够好

在把本地真实 embedding 真正接入之后，我发现一个很重要的现象：

- 检索阶段虽然已经能找到更语义化的内容
- 但如果不做过滤和后处理，极短残片 chunk 或泛化句子仍然可能扰乱最终结果

例如：

- 极短 chunk 会在向量检索中拿到异常高分
- 句子级答案选择可能被“语义相关但不够直接”的句子抢走

后来我通过：

- 过滤短 chunk
- 做 hybrid retrieval
- 强化 top1 chunk 对主回答的影响
- 为 chunks 预计算并缓存向量

才把最终规则版回答稳定拉回到 `chunk 1` 的定义句上。

## 今天遇到的困难，以及我是怎么解决的

### 1. 虚拟环境路径不匹配

一开始我在 WSL / Linux 终端里尝试使用：

- `.venv/bin/activate`

但项目里最早的虚拟环境其实是 Windows 风格的 `Scripts/activate`，导致激活失败。后来我重新在当前环境里建立了 Linux 风格的 `.venv`，才恢复成更标准的：

- `.venv/bin/activate`

这个问题让我更清楚地理解了：

- Windows 创建的虚拟环境
- Linux / WSL 创建的虚拟环境

在目录结构上是不一样的。

### 2. 缺少 `python-dotenv`

在运行 `rag_with_newapi.py` 时，我遇到了：

- `ModuleNotFoundError: No module named 'dotenv'`

这个错误让我意识到：

- 代码逻辑本身没问题
- 但 Python 运行环境里缺少依赖

也正是因为这次报错，我更清楚地理解了“依赖导入失败”和“API 调用失败”是两种不同的错误类型。

### 3. 自动回退为什么没有触发

我之前给 New API 版加了“失败回退到规则版回答”的逻辑，但后来发现：

- 当 `python-dotenv` 没有安装时，程序不会回退
- 而是直接报错退出

后来我理解到，原因是这次回退逻辑只包住了“接口请求阶段”，并没有把“依赖缺失”这种环境问题也一起吞掉。

这个设计虽然一开始看起来有点严格，但从学习角度看是合理的，因为它能让我更清楚地看到：

- 哪些问题属于代码逻辑
- 哪些问题属于环境配置

### 4. 纯 Jaccard 检索效果不理想

在检索实验中，我一开始以为把“简单交集打分”改成 Jaccard 就会自然更好，但实际测试后发现：

- 对于短 query 和长 chunk 的问答检索场景
- 纯 Jaccard 会把长 chunk 压分太厉害

这个问题不是简单的 bug，而是一个真正的检索设计问题。最终我通过加入：

- `query_coverage()`
- `Jaccard`

并组合成混合分数，才让检索结果重新变得更合理。

### 5. 本地真实 embedding 的依赖安装比预想中更复杂

在继续往 Day 7 后半段推进时，我开始尝试接入本地真实 embedding。原本以为只要安装 `sentence-transformers` 就行，但实际过程让我发现：

- 它会进一步依赖 `torch`
- 默认安装链还可能带出很多 CUDA / GPU 相关依赖

而我当前只是想做 CPU 本地 embedding，并不想在这个阶段处理 CUDA 环境。这让我意识到：

- 真正的向量检索，不只是模型概念问题
- 还会马上遇到依赖、平台和工程复杂度问题

这也让我更能区分：

- 学习一个检索思路
- 把它完整落到工程环境里

其实是两个层面的挑战。

## 我是如何调整推进策略的

在今天的过程中，我不再追求“一步到位把所有东西都切换成真实向量检索”，而是改成了更稳的推进方式：

1. 先把项目结构整理清楚
2. 先让规则版输出更可解释
3. 先把现有检索 baseline 做得更合理
4. 再为真实 embedding 预留结构
5. 最后再解决本地 CPU embedding 的依赖问题

这个调整让我更理解：

- 做项目时，节奏和顺序也很重要
- 有些问题不是“不会”，而是“现在先不该硬上”

## 今天学到的关键概念

### 1. 入口层和核心逻辑层

- `app.py` 更像前台
- `pipeline.py` 更像调度层
- 其它模块负责具体功能

### 2. 文本表示和相似度是两层问题

- `embedder.py` 负责“文本怎么表示”
- `retriever.py` 负责“两个表示怎么比较”

### 3. 归一化后的相似度不一定自动更好

Jaccard 理论上更公平，但在“短问题对长 chunk”场景下，可能会把本来相关的结果压低。

### 4. 可解释性是 RAG 的重要价值

一个好的 RAG 系统，不只是回答问题，还应该尽量告诉用户：

- 答案依据来自哪里

### 5. 工程升级要尽量平滑

今天我没有直接把项目硬切到真实 embedding，而是先保留 baseline，再为后续真实向量检索预留结构。这种做法更稳，也更容易持续迭代。

### 6. 真实 embedding 检索也需要后处理

今天后半段我进一步理解了：

- 用 embedding 取代 baseline，并不意味着可以取消所有后处理
- 真实检索系统仍然需要：
  - chunk 质量控制
  - 性能优化
  - hybrid 检索
  - 生成阶段和检索阶段之间的对齐

### 7. 向量缓存是很自然的下一步

今天我进一步体会到，真实 embedding 检索一旦接进来，马上就会遇到一个实际问题：

- 如果每次检索都重新给所有 chunks 做向量编码，速度会明显变慢

所以我后来增加了：

- `data/processed/chunk_embeddings.json`

把每个 chunk 的向量预先保存下来。这样后续检索时就可以：

- 只编码一次 query
- 直接读取 chunk 向量缓存

这让我更清楚地区分了：

- 在线计算
- 预计算 + 缓存

也让我感觉这个项目开始更像一个真正的检索系统，而不只是实验脚本。

1. 当前 chunk 切分仍然比较粗糙，会产生被硬切断的句子
2. 虽然已经有 chunk 向量缓存，但每个新进程首次运行时仍然要重新加载本地模型
3. 当前规则版回答仍然比较简单，句子筛选策略还有继续优化空间
4. 依赖安装和本地模型管理已经成为这个项目新的工程复杂度来源
5. 当前知识库规模仍然较小，检索实验的复杂度有限
