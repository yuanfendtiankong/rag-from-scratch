# Day 6 Notes

## 今天学习的内容

今天我继续优化了生成式 RAG 小项目，主要做了四件事：扩充知识库、增大 top-k 检索数量、加入简单的拒答机制，以及接入 New API 上的 gpt-5.2 模型来生成最终回答。通过这些改动，我进一步理解了“检索质量”“知识库内容”和“生成模型”会共同影响最终回答的参考性。

## 我今天做了什么

1. 扩充了 `knowledge.txt`，把原来很短的资料整理成更完整的 RAG 学习型知识库
2. 重新运行 `build_chunks.py`，基于新的知识库重新生成了 `chunks.json`
3. 把 `rag_with_newapi.py` 中的 top-k 从 3 改成了 5
4. 配置了 `.venv` 虚拟环境，并将 Python 版本切换到 3.11
5. 安装了调用接口所需的依赖，例如 `openai` 和 `python-dotenv`
6. 配置了 `.env` 文件，填入 New API 的 token、base_url 和模型名
7. 使用 New API 提供的 `gpt-5.2` 模型，完成了真正的生成式回答
8. 加入了简单的拒答阈值函数 `should_refuse`
9. 测试了知识库中有明确答案的问题，比如 `chunk_size 和 chunk_overlap 有什么作用`
10. 测试了知识库中没有明确答案的问题，比如 `Python 为什么适合机器学习`
11. 观察了模型在终端中的输出格式问题，例如出现 `**chunk_size**` 这样的 Markdown 加粗标记

## 我对今天优化的理解

### 1. 扩充知识库的重要性

之前我的知识库内容比较少，所以即使接入了真正的大模型，模型也没有足够的参考资料。今天扩充知识库后，我明显感觉到对于知识库中“确实有答案”的问题，模型回答得更完整、更有依据。

### 2. top-k 变大带来的影响

把 top-k 从 3 调到 5 之后，模型可以看到更多的上下文，所以有些问题的回答更完整了，比如不仅回答了 `chunk_size` 和 `chunk_overlap` 是什么，还补充了它们过大或过小时的影响。

但同时我也观察到，top-k 变大以后，也可能把一些低分 chunk 一起带进来，增加噪声。因此，top-k 不是越大越好，而是需要结合知识库和检索质量做实验调整。

### 3. 拒答机制的意义

今天加入拒答阈值后，当问题和知识库不相关时，系统不再强行编造一个答案，而是更倾向于明确表示“无法从当前资料中找到足够相关的内容”。这让我意识到，一个好的 RAG 系统不只是“能回答”，还要“知道什么时候不该回答”。

### 4. 接入 gpt-5.2 的意义

今天一个很重要的进展是，我不再只用规则去拼接句子，而是通过 New API 实际调用了 `gpt-5.2` 来生成回答。这意味着我的项目已经从“简单检索原型”进一步升级成了“真正接入大模型的生成式 RAG 初版”。

这一步也让我更清楚地看到：

- 检索质量不好时，即使模型很强，回答也可能不可靠
- 知识库内容足够好时，大模型可以把检索结果组织得更自然、更完整
- 大模型生成能力强，但仍然需要检索和拒答机制来约束

## 我的实验观察

### 问题 1：chunk_size 和 chunk_overlap 有什么作用

这个问题在知识库中有明确答案，所以系统可以检索到高分 chunks，并结合 `gpt-5.2` 生成较完整的回答。相比之前，回答中不仅解释了两个参数的基本作用，还提到了它们设置过大或过小时的影响。

### 问题 2：Python 为什么适合机器学习

这个问题在当前知识库里没有明确答案。加入拒答机制后，系统没有再像之前那样硬拼一个看起来相关的回答，而是更合理地说明无法从当前资料中确定答案。说明拒答机制是有效的。

### 关于调用 New API 的观察

我成功配置了 New API 的 token 和 base_url，并通过 OpenAI 兼容接口调用了 `gpt-5.2`。这说明我的项目已经不只是本地规则实验，而是真正接入了外部大模型服务。

### 关于 Markdown 输出的问题

我发现模型有时会输出像 `**chunk_size**` 这样的格式。这不是程序逻辑错误，而是因为模型默认会使用 Markdown 风格输出，而终端不会渲染 Markdown，所以星号会原样显示出来。

## 当前系统仍然存在的局限

1. 检索仍然基于简单字符片段重合，不是真正的 embedding 语义检索
2. top-k 虽然调大了，但仍然可能带入部分噪声
3. 拒答机制目前比较简单，只是基于 top1 分数和总分阈值
4. 模型输出格式有时会带 Markdown 标记，不够适合终端直接展示
5. 知识库虽然比之前丰富了，但规模仍然比较小
6. 当前生成依赖外部 API，成本和调用次数需要控制

## 今天的收获

今天我更清楚地理解了，RAG 的效果不只是由大模型本身决定，更依赖于知识库内容、检索质量、上下文组织方式以及拒答策略。同时，我也真正完成了从“简单检索系统”到“接入 gpt-5.2 的生成式 RAG 初版”的升级。
